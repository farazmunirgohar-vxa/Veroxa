## Future path structure

```
clients/{client_id}/raw/{yyyy}/{mm}/{uuid}-{safe_filename}
clients/{client_id}/processed/{yyyy}/{mm}/{uuid}-{safe_filename}
clients/{client_id}/menus/{yyyy}/{mm}/{uuid}-{safe_filename}
```

- `raw/`        ā€” what the client uploaded, unchanged.
- `processed/`  ā€” derived / compressed / cropped variants created later by
                  the team or an automated step.
- `menus/`      ā€” menu PDFs and menu image scans, kept separate because
                  validation and display rules differ.

`{safe_filename}` = lowercased, ASCII-only, non-ASCII stripped, spaces
replaced with `-`, original extension preserved.

## Allowed file types (initial)

| Kind | MIME types |
| --- | --- |
| Image | `image/jpeg`, `image/png`, `image/webp` |
| Video | `video/mp4`, `video/quicktime` |
| Menu  | `application/pdf` *(only inside `menus/`)* |

All other MIME types are rejected at the server boundary.

## Initial size limits

| Kind | Max size |
| --- | --- |
| Images | 15 MB |
| Videos | 250 MB |

## Future upload flow

1. Authenticated **client** picks files in the real (post-auth) media page.
2. Frontend calls a server endpoint with file metadata to **request upload
   intent**.
3. Server validates:
   - the caller's role is `client`,
   - the target `client_id` matches the caller's assignment,
   - MIME type is allowed,
   - file size is within the limit for its kind.
4. Server returns either a signed upload URL or accepts a small multipart
   stream on behalf of the client.
5. File is written to `clients/{client_id}/...` in `veroxa-client-media`.
6. A `media_assets` metadata row is created (see
   `002_media_assets_metadata_draft.sql`).
7. An `audit_logs` row is created (see
   `../write-draft/002_audit_log_draft.sql`).
8. The team review queue picks up the new asset (`review_status = 'new'`).

**The service role key MUST NOT be exposed to the frontend.** All bucket
writes are mediated by a server-side function.

## Future signed-URL strategy

- **Private bucket only.** No public URLs for client media.
- Server issues **short-lived** signed URLs (suggested default: 5 minutes)
  per asset, only after verifying the caller's role/assignment is allowed
  to view the asset.
- Signed URLs are never persisted; they're regenerated on demand.

## Future RLS / storage policy strategy

- **Client:** may upload only into their own `clients/{client_id}/...`
  path; may read only their own assets. Write must always create an
  `audit_logs` row.
- **Team:** may read assets for **assigned** clients (via
  `team_client_assignments`); may write `review_status`, `review_tags`,
  and team-side metadata only.
- **Operator:** may read all client assets; may approve / reject /
  archive.
- **Owner:** may read all client assets; mostly oversight, not daily
  writes.

## Out of scope for the first storage phase

- AI review / scoring (`quality_score`, `ai_review_summary`) is a later
  layer.
- Direct social publishing from media is a Priority 4 high-risk write
  (`docs/FIRST_WRITE_SURFACE_PLAN.md`) and is excluded.
- Malware scanning / content moderation must be revisited before going to
  real production traffic.
