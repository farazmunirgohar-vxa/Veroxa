## UUID strategy

Demo data uses string IDs such as `"client-mamadali-001"` and `"media-mamadali-003"`. These cannot be used directly as PostgreSQL UUID primary keys.

These seed files replace all string IDs with structured fixed UUID placeholders:

| Namespace | Range |
|-----------|-------|
| `clients` | `00000000-0000-0000-0000-000000000001` |
| `client_platforms` | `00000000-0000-0000-0001-000000000001` → `…004` |
| `onboarding_items` | `00000000-0000-0000-0002-000000000001` → `…006` |
| `media_assets` | `00000000-0000-0000-0003-000000000001` → `…010` |
| `post_slots` | `00000000-0000-0000-0004-000000000001` → `…008` |
| `posts` | `00000000-0000-0000-0005-000000000001` → `…007` |
| `weekly_reports` | `00000000-0000-0000-0006-000000000001` → `…002` |
| `monthly_reports` | `00000000-0000-0000-0007-000000000001` |
| `content_concepts` | `00000000-0000-0000-0008-000000000001` → `…007` |
| `draft_sets` | `00000000-0000-0000-0009-000000000001` → `…007` |
| `draft_variants` | `00000000-0000-0000-000a-000000000001` → `…021` |

Fixed UUIDs are used intentionally so that cross-file foreign key references are readable and predictable during dev seeding. In a real production seed workflow, `gen_random_uuid()` values would be generated and captured instead.

## Circular reference handling

`media_assets.used_in_post_id` and `post_slots.post_id` both reference `posts`, which in turn references `media_assets` and `post_slots`. This circular dependency is resolved in two passes inside `004_seed_posts_and_slots.sql`:

1. Insert `post_slots` with `post_id = NULL`
2. Insert `media_assets` with `used_in_post_id = NULL` *(already done in file 003)*
3. Insert `posts` referencing both
4. `UPDATE post_slots SET post_id = ...` to resolve the forward ref
5. `UPDATE media_assets SET used_in_post_id = ...` to resolve the circular ref
