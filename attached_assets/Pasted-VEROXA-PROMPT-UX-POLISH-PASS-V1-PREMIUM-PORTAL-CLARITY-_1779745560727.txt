UX GOAL

Veroxa should feel like a premium SaaS-style restaurant marketing operating system.

The user should immediately understand:

Client:
“Upload content, see upcoming posts, view reports, send requests.”

Team:
“Review media, create drafts, schedule content, handle reports and alerts.”

Operator:
“Monitor client health, approvals, quality, alerts, and system status.”

Owner:
“Watch revenue, growth, retention, critical risk, and system health.”

PART 2 — CLIENT PORTAL UX POLISH

Client portal must be the simplest.

Polish these pages:

src/pages/client-dashboard.tsx
src/pages/client-media.tsx
src/pages/client-calendar.tsx
src/pages/client-reports.tsx
src/pages/client-requests.tsx
src/pages/client-account.tsx

Client should NOT feel like they are managing an agency backend.

Client pages should use simple language:

Good:
- Upload media
- Upcoming posts
- This week’s performance
- Reports
- Requests
- Account

Avoid:
- AI workflow
- internal review queue
- operator escalation
- draft variants
- rejection logic
- backend/system terms
- too many metrics

Specific client polish:

1. Client Dashboard
Keep it focused on:
- This week snapshot
- Upload status
- Upcoming posts
- Latest report
- Recent activity

Do not overload with too many cards.

2. Client Media / Upload Media
Make this page feel like:
“Send us fresh restaurant content and we’ll handle the rest.”

Polish:
- Clear upload guidance
- Simple media status
- Simple content health reminder
- Friendly note about best photo/video types

Avoid:
- deep AI scoring
- internal rejection reasons
- too many production pipeline details

3. Client Calendar
Make upcoming posts easy to understand.
Use simple labels:
- Scheduled
- Posted
- Waiting on content

Avoid showing internal scheduling logic.

4. Client Reports
Weekly and monthly reports should feel premium and clear.
Reduce clutter if too many cards appear at once.

5. Client Requests
Make requests feel like a simple communication channel, not a task system.

6. Client Account
Make it clean:
- Business info
- Plan
- Contact
- Upload commitment
- Support / request access

Reduce excessive cards.

PART 4 — OPERATOR PORTAL UX POLISH

Operator portal should feel like oversight, not execution.

Polish visible Operator pages:

src/pages/operator-os.tsx
src/pages/client-health-command.tsx
src/pages/operator-alerts.tsx
src/pages/operator-report-approvals.tsx
src/pages/operator-media-library.tsx
src/pages/operator-team-oversight.tsx
src/pages/operator-system-status.tsx

Operator should see:
- Client health
- Critical and active alerts
- Monthly report approvals
- Failed/reschedule issues
- Team workload
- System status

Operator should NOT feel like:
- They approve every post
- They do daily production
- They manage every task manually

Polish rules:

1. Operator Command Center
Make this the main oversight page.
Prioritize:
- clients needing attention
- alerts
- monthly approvals
- team capacity
- system health

2. Client Health
Make health levels very clear:
- Healthy
- Caution
- Urgent
- Broken

3. Alerts
Separate:
- Critical
- Needs attention
- Resolved / low priority

4. Report Approvals
Make operator approval flow obvious but not cluttered.

5. Media Library
Make it oversight-level, not team-level.
Operator should see patterns and risk, not every small editing task.

6. System Status
Make it understandable:
- Demo only
- Connected
- Stubbed
- Future integration
- Not live yet

Avoid scary technical wording.
